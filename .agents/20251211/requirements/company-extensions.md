# 公司系统补充需求（2025-12-12）

## 推荐展示
- 工商系统上线后，前台首页或 `/company` 页面应展示对所有用户（登录与否）开放的公司推荐内容，包括最近注册、最近活跃、审核状态高的企业。
- 每条推荐需要呈现名称、类型、行业标签、法人/持有者姓名、状态与推荐评分，并可点击跳转到 `/company` 或后台详情。

## 权限与身份
- 只有管理员或拥有 `company.admin.*` 权限的角色才能在后台 `/admin/company/registry`、`/admin/company/applications`、`/admin/company/industries`、`/admin/company/types` 编辑所有企业与个体工商户信息。
- 持有者（OWNER）、法人（LEGAL_PERSON）等身份必须在流程中明确，并允许他们在前台 `/company/dashboard` 编辑自己名下公司的资料、制度与流程；其他玩家无此入口。
- 管理端与前端需要清晰告知“只有管理员可在后台统一编辑公司数据”，并把持有者/法人/制度的责任在 UI 中标注。

## 行业/类型/制度体系
- 系统应提供行业分类管理（可配置父级/颜色/图标、元数据）与公司类型管理（包含分类、说明、所需材料、JSON 配置），前台申请与推荐模块共同使用。
- 所有行业/类型配置应可在后台由具备 `company.admin.config` 权限的管理员维护，并会自动影响注册表单与推荐排序。
- 制度模块需要支持公司持有者或管理员录入章程、审批规则、内部流程等信息，并附带版本记录与状态（草稿/生效/暂停）。
- 推荐列表与流程事件可以联动，例如公司状态变为 ACTIVE/已入库时自动纳入近期活跃推荐、制度变更触发通知。

## 流程复用
- 流程框架必须高度可复用：定义流程模板、节点动作、角色、通知、超时规则后即可应用在公司审核、补件、制度审批、后续业务。
- 流程节点要支持绑定多个角色（如持有者、法人、管理员、审核员），并自动映射到公司事务（policy、workflowInstance、applicationStatus）。
- 每个申请、审核动作需记录审批记录、触发的推荐更新、代码注释与通知 hooks，方便后续业务调用。
